# BrainOS 项目总体需求

## 1. 项目目标

BrainOS 是一个面向企业内部资料的轻量级 AI 知识库。项目需要完整展示“资料导入、向量化、语义检索、基于依据的问答、来源追溯、基础后台管理”闭环，同时控制功能数量和部署复杂度，使其适合课程答辩与现场演示。

## 2. 用户与权限

| 角色 | 能力边界 |
| --- | --- |
| 管理员 | 使用全部业务功能，并管理用户、查看操作日志 |
| 普通用户 | 查看工作台，管理企业共享知识库和文档，使用 AI 问答及个人会话 |

系统不提供公开注册。管理员创建并维护普通用户账号。

## 3. 功能模块

1. 认证与权限：登录、刷新令牌、退出、角色访问控制。
2. 工作台：核心数量指标、七日问答趋势、最近文档。
3. 知识库：新建、编辑、查询、删除企业知识库。
4. 文档管理：上传、解析、切片、向量化、状态查询、重试、删除。
5. RAG 问答：限定知识库检索、流式回答、可靠性兜底、来源引用。
6. 会话管理：会话创建、历史查询、标题维护、删除。
7. 用户管理：管理员新建、编辑、启停用户。
8. 操作日志：记录并筛选关键管理与内容操作。

## 4. EARS 需求条目

### R1 认证与访问控制

- **End**：企业用户能够安全进入系统并只访问其角色允许的功能。
- **Approach**：通过用户名、密码、JWT 访问令牌、Redis 刷新令牌和 Spring Security 角色校验实现。
- **Rule**：系统仅允许状态为启用的账号登录；密码必须使用 BCrypt 摘要保存；管理员接口只允许管理员访问。
- **Constraint**：访问令牌有效期为 2 小时，刷新令牌有效期为 7 天；系统不开放自助注册。
- **Acceptance**：
  - A-R1-01：使用正确账号密码登录时，接口返回访问令牌、刷新令牌及用户角色，并进入工作台。
  - A-R1-02：密码错误或账号停用时，系统拒绝登录并显示不泄露账号存在性的错误信息。
  - A-R1-03：普通用户请求管理员接口时，系统返回 HTTP 403。
  - A-R1-04：退出登录后，对应刷新令牌立即失效。

### R2 工作台

- **End**：用户进入系统后能快速了解知识库运行情况。
- **Approach**：通过统计卡片、七日问答折线图和最近文档表格呈现。
- **Rule**：统计数据必须来自当前数据库状态，不使用硬编码演示数字。
- **Constraint**：页面仅展示知识库数、文档数、可用切片数、问答次数四项指标和一张趋势图。
- **Acceptance**：
  - A-R2-01：工作台显示四项统计数据，数值与数据库记录一致。
  - A-R2-02：七日问答趋势按日期聚合，并在无数据时显示明确空状态。
  - A-R2-03：最近文档显示名称、所属知识库、状态和更新时间。

### R3 知识库管理

- **End**：用户能够按业务主题组织企业资料。
- **Approach**：通过知识库列表与新建、编辑、删除操作实现。
- **Rule**：知识库对全部启用用户共享，名称全局不可重复；删除前必须二次确认。
- **Constraint**：知识库只包含名称、描述、创建人、创建时间和更新时间等必要字段。
- **Acceptance**：
  - A-R3-01：用户能够创建名称合法的知识库，并立即在列表中看到它。
  - A-R3-02：重复名称或空名称被前后端同时拦截，并显示字段级提示。
  - A-R3-03：删除知识库时，系统同步清理其文档、原文件、会话及 Chroma 向量。

### R4 文档导入

- **End**：用户能够将常见企业文档导入指定知识库。
- **Approach**：通过浏览器上传、后端本地文件存储、文本解析和异步处理实现。
- **Rule**：只接受 PDF、DOCX、TXT、Markdown；同时校验扩展名和 MIME 类型；文件名必须清理后保存。
- **Constraint**：单文件不超过 20MB；不支持扫描件 OCR；同一知识库内根据 SHA-256 文件哈希阻止重复导入。
- **Acceptance**：
  - A-R4-01：合法文件上传后生成文档记录，状态依次经过“解析中、向量化中、可用”或进入“失败”。
  - A-R4-02：不支持的类型、超限文件和重复文件在写入向量库前被拒绝，并显示具体原因。
  - A-R4-03：扫描型 PDF 未提取到文本时状态为失败，失败原因为“未提取到可用文本”。
  - A-R4-04：失败文档可以重试，重试成功后状态变为可用且不产生重复切片。

### R5 Chroma 向量索引

- **End**：导入资料能够被语义检索并按知识库隔离。
- **Approach**：使用 Spring AI 文档模型和切分器、千问 Embedding 模型、Chroma Vector Store 实现。
- **Rule**：每个切片必须携带 `knowledgeBaseId`、`documentId`、`fileName`、`pageNumber`、`chunkIndex` 元数据。
- **Constraint**：Chroma 作为演示向量库；MySQL 不重复保存切片正文；Embedding 维度在同一集合内保持一致。
- **Acceptance**：
  - A-R5-01：可用文档的切片数量写回 MySQL，并能在 Chroma 中按文档 ID 检索到等量记录。
  - A-R5-02：检索请求只能返回所选知识库的切片。
  - A-R5-03：删除文档后，Chroma 中不再存在对应文档 ID 的记录。

### R6 RAG 流式问答

- **End**：用户能够基于企业资料获得可追溯的回答。
- **Approach**：将用户问题向量化，从 Chroma 取相似度最高的 5 个片段，组装上下文后调用千问或 DeepSeek，并通过 SSE 返回结果。
- **Rule**：千问是默认聊天模型；DeepSeek 是可切换聊天模型；Embedding 始终使用千问；回答必须绑定所选知识库。
- **Constraint**：模型密钥只从环境变量读取；前端不接触供应商密钥；系统不训练或微调模型。
- **Acceptance**：
  - A-R6-01：用户选择知识库并提问后，页面持续接收并渲染流式回答。
  - A-R6-02：回答完成后展示实际使用的文件名、页码和命中片段。
  - A-R6-03：没有达到相关度阈值的资料时，系统回答“当前知识库中未找到可靠依据”，且不生成无依据内容。
  - A-R6-04：聊天模型不可用时保留用户问题，显示可重试错误，不保存空 AI 消息。

### R7 会话管理

- **End**：用户能够继续和回顾自己的知识问答。
- **Approach**：通过会话列表、消息历史和会话删除实现。
- **Rule**：普通用户只能访问自己的会话；会话固定关联一个知识库和当前选择的聊天模型。
- **Constraint**：会话标题取首条问题的精简文本，允许用户修改。
- **Acceptance**：
  - A-R7-01：刷新页面后，用户能够重新打开历史会话并看到完整问题、回答与引用。
  - A-R7-02：访问其他用户会话时返回 HTTP 403 或 HTTP 404。
  - A-R7-03：删除会话后，其消息与引用数据一并删除。

### R8 用户管理

- **End**：管理员能够维护演示系统用户。
- **Approach**：通过用户表格和新建、编辑、启停表单实现。
- **Rule**：用户名唯一；至少保留一个启用的管理员；普通用户不能访问该模块。
- **Constraint**：角色仅有管理员和普通用户，不实现部门、菜单和数据权限树。
- **Acceptance**：
  - A-R8-01：管理员能够新建普通用户，新用户可使用初始密码登录。
  - A-R8-02：停用用户后，该用户无法刷新令牌或重新登录。
  - A-R8-03：系统拒绝停用最后一个启用的管理员。

### R9 操作日志

- **End**：答辩演示能够说明关键操作具备可追溯性。
- **Approach**：通过后端审计服务记录登录、知识库变更、文档上传/重试/删除和用户管理操作。
- **Rule**：日志包含操作人、操作类型、目标类型、目标 ID、结果、时间和必要摘要，不记录密码、令牌或模型密钥。
- **Constraint**：只实现列表、分页以及按用户、操作类型、时间筛选，不实现日志导出和告警。
- **Acceptance**：
  - A-R9-01：完成受审计操作后，可以在日志页查到对应记录。
  - A-R9-02：筛选条件改变时返回匹配的分页结果。
  - A-R9-03：日志详情不包含任何敏感凭据。

### R10 可部署性与质量

- **End**：项目能够在答辩电脑上稳定启动、演示和复验。
- **Approach**：使用 Docker Compose 管理 MySQL、Redis、Chroma，提供环境变量模板、初始化数据、自动化测试和操作说明。
- **Rule**：禁止使用云函数或 Serverless 平台；项目配置不得提交真实密钥。
- **Constraint**：目标浏览器为当前版 Chrome 和 Edge，桌面视口为 1024px 与 1440px；不开发移动端页面。
- **Acceptance**：
  - A-R10-01：按 README 步骤可启动全部依赖、后端和前端，并使用默认管理员进入系统。
  - A-R10-02：自动化测试覆盖认证、权限、文档索引、RAG 引用和核心页面交互。
  - A-R10-03：Playwright 能完成“登录、新建知识库、上传文档、等待索引、提问、查看引用”的主链路。
  - A-R10-04：1024px 和 1440px 视口下无主要内容裁切、遮挡或横向滚动。
  - A-R10-05：启动后可以通过 Swagger UI 查看与调用自有后端 API，且 OpenAPI 文档不暴露模型密钥或其他敏感配置。

## 5. 非功能要求

- **可用性**：所有异步操作显示进行中、成功、失败或空状态；错误信息说明用户可采取的下一步。
- **性能**：本地演示环境中，非 AI 列表和表单接口的 P95 响应时间目标低于 500ms；模型网络正常时，流式回答首个内容事件目标在 10 秒内返回。
- **安全性**：接口参数统一校验；SQL 使用 MyBatis-Plus 参数绑定；上传路径不可由客户端直接控制；管理接口执行服务端授权。
- **可维护性**：后端按业务功能分包，前端按功能模块组织；控制器不承载业务逻辑；公共响应和异常结构统一。
- **可测试性**：每条 Acceptance 在模块 `tasks.md` 中映射到自动化测试或明确的验收步骤。

## 6. 明确不做

- 微信小程序、鸿蒙、iOS、Android 或任何移动端适配。
- 扫描件 OCR、图片理解、音视频解析、Excel 数据问答。
- 多租户、组织架构、复杂审批、SSO、LDAP、细粒度数据权限。
- 微服务、消息队列、分布式任务调度、云函数、Serverless。
- 模型训练、微调、Agent 工作流、联网搜索、语音交互。
- 对象存储、日志导出、复杂告警和商业计费。
