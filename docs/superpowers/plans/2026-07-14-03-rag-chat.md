# BrainOS Phase 3 RAG Chat Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver knowledge-base-scoped RAG conversations with Qwen/DeepSeek routing, reliable-evidence fallback, streamed SSE answers, persisted history, and source citations.

**Architecture:** Separate retrieval, prompt construction, model routing, streaming protocol and persistence behind small interfaces. Persist a user question before generation, persist an AI message only after successful completion, and keep citation data derived from the exact retrieved chunks used in the prompt.

**Tech Stack:** Spring AI 1.1.8, Spring MVC SSE/Flux bridge, Qwen Embedding, Qwen Chat, DeepSeek Chat, Chroma, MyBatis-Plus, Vue 3, Pinia, Marked + DOMPurify, Vitest, Playwright.

## Global Constraints

- Qwen is the only embedding provider; Qwen is default chat and DeepSeek is selectable chat.
- Retrieval is filtered by `knowledgeBaseId`, Top 5, with a configurable similarity threshold.
- If no result reaches the threshold, return `当前知识库中未找到可靠依据` without calling a chat model.
- SSE event names are exactly `start`, `delta`, `citations`, `done`, `error`.
- A user can read, rename and delete only their own conversations.
- Do not persist an empty or partial AI response as a successful assistant message.

- [ ] **Step 3: Implement retrieval and planning**

Embed the question with the existing Qwen embedding port, call Chroma with `knowledgeBaseId == selected`, `topK=5`, threshold from configuration, sort descending, and convert metadata into immutable citation candidates. Treat missing/invalid metadata as an indexing error rather than leaking incomplete citations.

```java
public RagAnswerPlan plan(long knowledgeBaseId, String question) {
    List<CitationCandidate> candidates = retriever.retrieve(knowledgeBaseId, question, 5, threshold);
    if (candidates.isEmpty()) return RagAnswerPlan.fallback("当前知识库中未找到可靠依据");
    if (candidates.stream().anyMatch(candidate -> candidate.knowledgeBaseId() != knowledgeBaseId)) {
        throw new VectorMetadataException("检索结果知识库不一致");
    }
    return RagAnswerPlan.grounded(candidates);
}
```

- [ ] **Step 3: Implement SSE orchestration**

Save the user message first. For fallback emit `start`, one `delta`, empty `citations`, `done` without model invocation. For model success accumulate deltas, then persist exactly one assistant message and emit citations/done. On error emit one sanitized `error` event and persist no assistant message. Cancel provider subscription when the client disconnects.

```java
public Flux<ChatStreamEvent> ask(long sessionId, long userId, String question) {
    ChatSession session = sessions.getOwned(sessionId, userId);
    messages.saveUser(sessionId, question);
    RagAnswerPlan plan = planning.plan(session.knowledgeBaseId(), question);
    if (plan.isFallback()) return Flux.just(
        ChatStreamEvent.start(), ChatStreamEvent.delta(plan.fallback()),
        ChatStreamEvent.citations(List.of()), ChatStreamEvent.done());
    StringBuilder answer = new StringBuilder();
    return router.stream(session.model(), prompts.create(question, plan.citations()))
        .map(delta -> { answer.append(delta); return ChatStreamEvent.delta(delta); })
        .startWith(ChatStreamEvent.start())
        .concatWith(Flux.defer(() -> {
            messages.saveAssistant(sessionId, answer.toString(), plan.citations());
            return Flux.just(ChatStreamEvent.citations(plan.citations()), ChatStreamEvent.done());
        }))
        .onErrorResume(error -> Flux.just(ChatStreamEvent.error(errors.userMessage(error))));
}
```
