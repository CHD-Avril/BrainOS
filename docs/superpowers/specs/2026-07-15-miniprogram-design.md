# BrainOS 微信小程序客户端设计

## 1. 目标与范围

在现有 BrainOS 仓库中新建独立的 `miniprogram/` 客户端，使用 uni-app CLI、Vue 3、TypeScript 和 Pinia 构建微信小程序，并复用 Spring Boot 的 `/api/v1/**` 接口。现有 `frontend/` PC Web 与 `backend/` 后端保持独立，两端共享同一套后端业务和数据。

首版范围包括：

- 用户名、密码登录，JWT 访问令牌与刷新令牌续期。
- 知识库列表、详情、新建、编辑和删除。
- 文档列表、选择、上传、状态查看、失败重试和删除。
- 问答会话列表、新建、重命名、删除、历史消息与流式问答。
- 用户信息与退出登录。

首版不包括仪表盘、用户管理、审计日志、微信授权登录、独立后端小程序协议或客户端伪分页。

## 2. 技术方案

采用 uni-app CLI + Vue 3 + TypeScript + Pinia，通过 pnpm 管理依赖，并提供 Vitest 单元测试、TypeScript 类型检查和微信小程序构建命令。选择该方案是因为它与现有 Vue 3 PC Web 技术栈最接近，同时可以针对微信平台封装 `uni.request`、`uni.uploadFile` 和分块响应能力。

不采用 HBuilderX 专用工程，以避免构建和测试流程依赖图形化 IDE；不采用原生微信小程序，以避免页面、状态管理和测试体系脱离现有 Vue 技术栈。

仓库结构为：

```text
frontend/       PC Web
miniprogram/    微信小程序
backend/        共用 Spring Boot API
```

`miniprogram/src/` 内按职责划分：

```text
src/
├─ api/          通用请求、令牌刷新、文件上传、SSE 与业务 API
├─ stores/       登录、知识库、文档和问答状态
├─ pages/        登录、知识库、文档、问答和我的页面
├─ components/   可复用业务组件
├─ config/       环境与 API 地址配置
└─ types/        API 契约与平台类型
```

每个模块只承担一个职责：普通 JSON 请求、令牌刷新、文件上传和流式响应分别封装，业务页面不直接调用微信底层网络 API。

## 3. 导航与页面

登录成功后的底部导航为“知识库”“AI 问答”“我的”。文档管理从知识库详情进入，不单独占用底部导航入口。

### 3.1 登录页

- 提供用户名和密码输入。
- 复用 `POST /api/v1/auth/login`。
- 启动时恢复本地会话；访问令牌失效时优先刷新，刷新失败后才返回登录页。

### 3.2 知识库与文档

- 知识库页展示卡片列表，支持下拉刷新、新建、编辑和删除。
- 删除知识库前必须二次确认。
- 知识库详情展示名称、描述、文档总数和可用文档数，并提供文档入口。
- 文档页展示 `PARSING`、`INDEXING`、`READY`、`FAILED` 四种状态。
- 文件通过微信会话文件选择能力选择，首版允许 PDF、DOCX、TXT、Markdown，上传前校验扩展名和 20MB 大小限制。
- 上传使用小程序上传 API，multipart 字段名固定为后端要求的 `file`。
- 失败文档支持重试，所有文档支持删除并进行二次确认。

### 3.3 AI 问答

- 问答首页展示当前用户的历史会话。
- 新建会话时选择知识库和 `QWEN` 或 `DEEPSEEK` 模型。
- 支持会话重命名和删除。
- 对话页加载历史消息，提交问题后逐步渲染回答。
- 回答完成后展示可展开的引用来源，包括文件名、片段、页码与相关度信息。
- 用户可以停止生成；停止后保留已收到的回答内容。
- 流式失败时保留用户问题并提供重试。

### 3.4 我的

- 展示用户名、显示名、角色和当前环境信息。
- 退出登录时调用现有注销接口；无论远端注销是否成功，都清除本地会话。
- 管理员与普通用户共享上述用户端页面，小程序不提供管理后台入口。

## 4. API 与鉴权

普通业务请求由 `api/request.ts` 统一封装：

1. 从本地会话读取 access token 并注入 `Authorization: Bearer <token>`。
2. 调用以 `/api/v1` 为前缀的现有接口。
3. 解析后端 `ApiResponse<T>` 信封。
4. 将错误统一转换为包含 `code`、`message` 和 `traceId` 的客户端错误。

收到 401 时，客户端使用 refresh token 调用现有刷新接口。同一时刻只允许一个刷新请求；其他失败请求等待刷新结果，成功后各自重试一次。刷新失败时清除本地会话并返回登录页，防止刷新循环。

本地存储的会话包含 access token、refresh token 和用户摘要。读取时必须进行结构校验；损坏、缺字段或角色非法的数据直接清除。客户端不得包含微信 AppSecret、AI Key、数据库凭证、JWT 签名密钥或其他后端密钥。

## 5. 文件上传

客户端通过微信会话文件选择能力获取临时文件路径，再通过 `uni.uploadFile` 上传到：

```text
/api/v1/knowledge-bases/{knowledgeBaseId}/documents
```

上传请求设置 JWT 请求头，字段名为 `file`。由于小程序上传接口返回的响应体是字符串，上传模块负责解析 JSON 信封并复用普通请求的错误模型。

客户端的扩展名和 20MB 检查用于快速反馈，Spring Boot 和 Nginx 限制仍是最终安全边界。取消选择不显示错误；网络失败或服务端失败保留当前页面并允许原地重试。

## 6. SSE 流式问答

小程序继续调用现有接口，不新增后端问答协议：

```text
POST /api/v1/chat/sessions/{sessionId}/messages/stream
Accept: text/event-stream
```

微信端请求开启 `enableChunked`，并通过 `RequestTask.onChunkReceived` 接收 `ArrayBuffer` 分块。SSE 模块必须：

- 使用增量 UTF-8 解码，正确处理中文字符的字节跨 chunk 情况。
- 保存未完成文本，正确处理 SSE 事件边界跨 chunk 的情况。
- 支持一个 chunk 包含多个事件。
- 识别 `start`、`delta`、`citations`、`done`、`error` 事件。
- 在 `done` 后完成本次请求，在 `error` 后显示服务端消息。
- 通过 `RequestTask.abort()` 停止生成，并保留已收到文本。

若运行环境无法监听分块响应，则等待同一个 HTTP 请求结束，并将完整响应按 SSE 规则一次性解析展示，不调用第二套接口。

代理层必须保持 SSE 路径的 `proxy_buffering off` 和 `proxy_cache off`。实施时需要移除或调整现有 `chunked_transfer_encoding off`，并在微信开发者工具和真机中验证 `onChunkReceived` 能及时触发。

## 7. 环境与部署

小程序通过构建环境配置 API 根地址。仓库提供开发和生产配置示例，但不提交真实域名、AppID、证书或密钥。

生产要求：

- API 使用有效证书的 HTTPS 公网域名，不能使用 localhost、IP 地址或自签名证书。
- 同一 API 域名加入微信公众平台的 `request` 和 `uploadFile` 合法域名。
- Nginx 将 `/api` 转发至 Spring Boot，并保持现有 25MB 请求体上限与 SSE 专用超时。
- PC Web 继续由原路径提供服务，不受小程序构建和发布影响。

正式联调前，开发者在本地未提交的环境文件中填写实际 AppID 与 API 域名，再将构建产物导入微信开发者工具。

## 8. 异常处理

- 表单校验错误在提交前就地提示。
- API 业务错误优先展示后端安全消息，并保留 `traceId` 供排障。
- 断网、超时和上传失败提供原地重试，不自动重复执行可能已成功的创建或删除操作。
- 401 仅自动刷新和重试一次；刷新失败统一退出。
- SSE 网络中断与服务端 `error` 事件分别提示，已接收内容和用户问题均保留。
- 文档轮询或刷新失败不清空已有列表，避免短暂网络故障造成空页面误导。

## 9. 测试与验收

自动化测试覆盖：

- 会话持久化、损坏数据清理和退出登录。
- JWT 注入、401 自动刷新、并发请求只刷新一次和刷新失败退出。
- `ApiResponse<T>` 与错误响应解析。
- 上传响应解析、扩展名和 20MB 限制。
- SSE 增量解析：中文 UTF-8 字节拆分、事件拆分、单 chunk 多事件、完整响应兜底和错误事件。
- 知识库、文档和会话状态的成功、失败与刷新行为。
- 登录、知识库 CRUD、上传与重试、会话 CRUD、停止和重试问答等关键页面交互。

持续验证命令为：

```bash
cd miniprogram
pnpm test
pnpm typecheck
pnpm build:mp-weixin
```

微信开发者工具和真机验收包括：

1. 在 HTTPS 域名下完成登录和令牌刷新。
2. 上传真实支持格式的文档，并观察状态变为可用。
3. 问答内容逐步出现，中文无乱码，引用信息正确。
4. 主动停止流式请求后页面仍可继续操作。
5. 禁用分块能力后，同一 SSE 请求结束时一次性展示回答。
6. PC Web 现有测试、类型检查和构建继续通过。

## 10. 完成标准

- `miniprogram/` 可以独立安装依赖、运行测试、类型检查并构建微信小程序产物。
- 登录、知识库 CRUD、文档管理、会话管理和问答均复用现有 `/api/v1/**` 契约。
- 微信端支持分块流式回答和同请求完整响应兜底。
- Spring Boot 不增加小程序专用问答协议。
- 配置正式域名和 AppID 后，可以导入微信开发者工具完成联调与发布准备。
- 现有 PC Web 保持可用且验证通过。
